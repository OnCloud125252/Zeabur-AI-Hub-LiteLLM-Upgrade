## 摘要

此 PR 解決了 **Gemini 3** 與不原生支援 Gemini「thought signatures」的聊天完成客戶端（如 LangChain 的 ChatOpenAI）之間的相容性問題。

### 問題

Gemini 3 在其回應中的工具呼叫內返回 thought signatures，並期望客戶端在後續請求中發回 thought signature。如果未提供 thought signature，Gemini 會返回 400 錯誤。許多聊天完成客戶端（例如 LangChain 的 ChatOpenAI）不會自動處理此情況。

**參考資料：** [Gemini Thought Signatures 文件](https://ai.google.dev/gemini-api/docs/thought-signatures)

### 解決方案

將 Gemini thought signatures 儲存在 **tool call ID** 中。這確保聊天完成客戶端會在聊天歷史記錄中始終傳回 tool call ID。LiteLLM 然後從 tool call ID 中提取 thought signature 並將其發送給 Gemini。

**注意：** 這僅適用於聊天完成，不適用於 Responses API。

## 技術細節

### Thought Signature 格式

Tool call ID 被修改為以特定格式包含 thought signature：

```
call_<uuid>__thought__<signature>
```

範例：

```
call_00d09e4f03654b668abe3e361f1e__thought__CsoDAePx/140cmKGChx3tcqSw7eWw+UgW8B...
```

## 影響

此變更透過 LiteLLM 的代理程式實現了與 Gemini 3 模型的適當工具呼叫，即使在使用不原生支援 Gemini thought signature 協定的客戶端時也是如此。這改善了與以下項目的相容性：

- LangChain 的 ChatOpenAI
- OpenAI 相容客戶端
- 任何遵循標準 tool call ID 模式的聊天完成客戶端
