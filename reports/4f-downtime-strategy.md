## 執行摘要

本文件比較 Blue-Green 部署與停機升級兩種方案的停機時間，提供時序圖與減少 downtime 的策略，並包含 Phase 3 實際驗證數據。

| 指標 | 數值 |
|------|------|

## 1. 三種方案比較

| 項目 | 方案 A：Blue-Green | 方案 B：停機升級 | 方案 C：Canary |
|------|-------------------|-----------------|---------------|
| **預估停機時間** | **< 30 秒** | **5-10 分鐘** | **0 秒** |
| 適用環境 | 有 LB 或 K8s | 單機 Docker | K8s + Service Mesh |
| 複雜度 | 中 | 低 | 高 |
| 資源需求 | 需同時運行兩個實例 | 僅需一個實例 | 需 K8s + Istio/Linkerd |
| 回滾速度 | < 30 秒（切 LB） | 5-10 分鐘（重啟舊版） | < 5 秒（調權重） |
| 風險 | 低（可即時切回） | 中（需停機回滾） | 最低（漸進驗證） |
| **推薦度** | **推薦** | 可接受 | 最推薦（如有基礎設施） |

## 2. Blue-Green 部署時序圖

```
時間軸 (分鐘)
─────────────────────────────────────────────────────────────────
t+0    [準備] 事先 docker pull 新映像（可提前數小時完成）

t+0    [DB 遷移] 執行 Phase A + B SQL（舊版仍在運行中）
       ├─ Phase A SQL: ~5 秒
       └─ Phase B SQL: ~10 秒

t+1    [部署] 啟動新版本實例（port 4001）
       └─ 等待 health check 通過 (~40 秒 start_period)

t+2    [驗證] 對新實例執行快速測試
       ├─ health check: 1 秒
       ├─ model list: 1 秒
       └─ chat completion: 2 秒

t+3    [切換] Load balancer 流量切換 ◄── 唯一停機點（< 30 秒）

t+3.5  [監控] 觀察新實例 5 分鐘

t+8.5  [清理] 確認無問題後停止舊實例
─────────────────────────────────────────────────────────────────
總耗時: ~9 分鐘（停機 < 30 秒）
```

**關鍵**：DB 遷移在舊版運行中執行（可加性變更），唯一停機點是 LB 流量切換。

## 3. Canary 部署（最安全）

**預估停機時間**：0 秒（無縫漸進）
**適用條件**：有 Kubernetes + Service Mesh（Istio/Linkerd）或具備流量分割能力的 LB

### 實施步驟

1. **初始階段（5% 流量）**
   - 部署新版本 replica: 1
   - 設定流量權重: 舊版 95%, 新版 5%
   - 監控 10-15 分鐘

2. **驗證階段（20% 流量）**
   - 檢查錯誤率、延遲指標
   - 如正常，擴大至 20% 流量
   - 監控 10 分鐘

3. **擴大階段（50% 流量）**
   - 繼續監控關鍵指標
   - 觀察是否有邊緣案例錯誤
   - 監控 10 分鐘

4. **全面切換（100% 流量）**
   - 確認所有指標正常
   - 切換 100% 流量至新版本
   - 保留舊版本 30 分鐘後銷毀

### 快速回滾

任一階段發現問題，可立即將流量權重調回 0%，無需重啟服務。

## 4. 停機升級時序圖

```
時間軸 (分鐘)
─────────────────────────────────────────────────────────────────
t+0    [準備] 事先 docker pull 新映像

t+0    [停機開始] docker compose down ◄── 停機開始
       └─ 服務停止: ~10 秒

t+0.5  [備份] pg_dump（如尚未備份）
       └─ 備份時間: ~30 秒（視資料量）

t+1    [遷移] 執行 SQL 遷移（或讓 Prisma 自動遷移）
       └─ 遷移時間: ~15 秒

t+1.5  [設定] 更新 docker-compose.yml
       └─ 修改 image + healthcheck: ~30 秒

t+2    [啟動] docker compose up -d
       └─ 啟動 + health check: ~40 秒

t+3    [驗證] 基本功能測試
       └─ 測試時間: ~30 秒

t+3.5  [恢復] 確認服務正常 ◄── 停機結束
─────────────────────────────────────────────────────────────────
總耗時: ~4 分鐘（停機 ~3.5 分鐘）
```

## 5. 減少 Downtime 的 5 項策略

### 5.1 策略 1：提前拉取映像

```bash
# 提前數小時或前一天執行
docker pull docker.litellm.ai/berriai/litellm:v1.81.12-stable.1
```

映像拉取可能需要 1-5 分鐘（視網路），提前完成可避免佔用升級窗口。

### 5.2 策略 2：提前執行 DB 遷移

由於所有 schema 變更都是可加性的：

- 新增表 → `CREATE TABLE IF NOT EXISTS`
- 新增欄位 → `ADD COLUMN IF NOT EXISTS` + nullable/default
- 新增索引 → `CREATE INDEX IF NOT EXISTS`

可以在 v1.79.0 **仍在運行時**安全執行遷移。v1.79.0 會忽略不認識的表和欄位。

### 5.3 策略 3：使用 Blue-Green 部署

同時運行新舊版本，僅在 LB 切換時產生極短停機（< 30 秒）。這是最推薦的方式。

### 5.4 策略 4：選擇低流量時段

建議在以下時段執行升級：

- 深夜 / 凌晨（使用者流量最低）
- 週末（如業務允許）
- 維護窗口（如有既定排程）

### 5.5 策略 5：準備自動回滾腳本

```bash
#!/bin/bash
# rollback.sh - 一鍵回滾腳本

echo "開始回滾至 v1.79.0..."

# 停止新版本
docker compose down

# 還原設定
cp docker-compose.yml.backup docker-compose.yml

# 啟動舊版本
docker compose up -d

# 等待 health check
sleep 45
curl -f http://localhost:4000/health/liveliness && echo "回滾成功" || echo "回滾失敗，需手動介入"
```

## 6. 實際驗證數據

Phase 3 遠端測試的實際耗時：

| 任務 | 預估時間 | 實際耗時 |
|------|----------|----------|
| Docker 環境建置 | 30 分鐘 | **2 分鐘** |
| v1.79.0 基準線建立 | 30 分鐘 | **4 分鐘** |
| 資料庫遷移 | 1 小時 | **2 分鐘** |
| v1.81.12 升級部署 | 30 分鐘 | **2 分鐘** |
| 功能迴歸測試 | 1 小時 | **2 分鐘** |
| 效能比對測試 | 30 分鐘 | **8 分鐘** |
| 回滾測試 | 30 分鐘 | **2 分鐘** |
| **總計** | **4 小時** | **約 23 分鐘** |

> **分析**：實際升級時間遠低於預估。其中「升級部署」僅需 2 分鐘（含映像啟動 + health check），證實停機時間可以控制在很短的範圍內。

## 7. Zeabur 平台適用建議

Zeabur 平台基於 Docker 容器部署，以下建議針對此環境：

### 7.1 推薦方案

使用 **Blue-Green 部署**（方案 A），利用 Zeabur 平台的容器管理能力：

1. 部署新版本容器（不接收流量）
2. 健康檢查通過後切換流量
3. 觀察一段時間後移除舊容器

### 7.2 注意事項

| 項目 | 說明 |
|------|------|
| 映像來源 | 需確認 Zeabur 平台可存取 `docker.litellm.ai` |
| Health check | 更新為 `python3 urllib` 版本 |
| 環境變數 | 在 Zeabur 控制台更新（不需改代碼） |
| DB 遷移 | 建議手動執行 SQL + 設定 `DISABLE_SCHEMA_UPDATE=true` |
| 回滾 | 保留舊容器 image 快照，可快速切回 |

### 7.3 建議時間窗口

- **準備工作**（可提前完成）：拉映像、備份 DB、準備設定 — 約 30 分鐘
- **實際升級**：DB 遷移 + 部署 + 驗證 — 約 **5-10 分鐘**
- **監控期**：升級後觀察 — 建議 **30 分鐘**
- **服務不可用時間**（Blue-Green）：**< 30 秒**
